# Codex Workflow

## 工作原则

OpenAthor 准备用 Codex 完成开发，但 Codex 在前期阶段应先作为 PM、技术写作者和架构审查者使用，而不是直接写代码。

在用户明确说可以进入实现前，Codex 应优先推进：

- 产品边界澄清
- 用户故事拆解
- 文档结构维护
- 协议和 CLI 合约设计
- 验收样例设计
- 风险和开放问题收敛

## 每次迭代的推荐流程

1. 确认当前目标属于产品、协议、CLI、skill、验收还是实现
2. 读取对应顶层路由文档
3. 读取最相关的子文档
4. 更新对应文档
5. 如新增主题，同步更新路由页
6. 汇总改变和仍未解决的问题

## 代码闸门

在 `docs/pre-development/readiness-checklist.md` 的关键项未完成前，Codex 不应开始写产品代码。

可以创建和维护以下非产品代码资产：

- `.codex/skills/` 下的项目协作 skill
- 文档
- 示例小说夹具设计文档
- 伪代码或 CLI 合约示例

## 任务追溯

后续每个实现任务都应能追溯到：

- 一个用户故事
- 一个 CLI 合约
- 一个验收样例
- 一个文档位置

如果不能追溯，说明任务可能太早或范围不清。
